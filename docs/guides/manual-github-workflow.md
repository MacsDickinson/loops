# Manual GitHub Workflow Guide

**Version:** 1.0
**Last Updated:** 2026-03-07
**Status:** Beta MVP (v1.0)

## Overview

Discovery Loop Coach v1.0 exports specifications in Markdown format. This guide explains how to manually create GitHub Pull Requests or Issues from your exported specs.

**Why manual workflow?** Automated PR creation (MAC-33) is deferred to v1.1 to accelerate beta launch. This manual process is simple and works immediately.

## Quick Start

1. Complete your specification in Discovery Loop Coach
2. Export spec as Markdown (⌘+E or "Export" button)
3. Spec is copied to clipboard automatically
4. Navigate to GitHub → Create PR or Issue
5. Paste spec into description field
6. Submit

**Time:** ~30 seconds per export

## Detailed Instructions

### Step 1: Export Your Spec

**In Discovery Loop Coach:**

1. Complete your discovery dialogue
2. Review the generated specification
3. Click **Export** button (or press ⌘+E)
4. Spec is automatically copied to clipboard in Markdown format
5. Confirmation message: "✓ Spec copied to clipboard"

**Export Format:**

```markdown
# [Feature Title]

## Overview
[Description from dialogue]

## Requirements

### Functional Requirements
1. [Requirement 1]
2. [Requirement 2]
...

### Non-Functional Requirements
1. [NFR 1]
2. [NFR 2]
...

## Acceptance Criteria

### Feature: [Feature Name]

**Scenario:** [Scenario name]
- **Given** [precondition]
- **When** [action]
- **Then** [expected result]

[Additional scenarios...]

---
*Generated by Discovery Loop Coach*
*Spec ID: [spec-id] | Created: [timestamp]*
```

### Step 2A: Create GitHub Pull Request

**Use case:** When you have code ready and want spec as PR description

1. Navigate to your GitHub repository
2. Create a new branch (or switch to feature branch)
   ```bash
   git checkout -b feature/your-feature-name
   ```
3. Make your code changes and commit
4. Push branch to GitHub
   ```bash
   git push origin feature/your-feature-name
   ```
5. GitHub will prompt "Compare & pull request" → Click it
6. In PR description field: **Paste** (⌘+V) your exported spec
7. Review title and branch targets
8. Click **Create pull request**

**Result:** PR created with your spec as the description

### Step 2B: Create GitHub Issue

**Use case:** When you want to track the work before coding

1. Navigate to your GitHub repository
2. Go to **Issues** tab
3. Click **New issue**
4. In issue description field: **Paste** (⌘+V) your exported spec
5. Add title (suggestion: use spec feature title)
6. Add labels (optional): `specification`, `discovery-loop`
7. Assign to team member (optional)
8. Click **Create issue**

**Result:** Issue created with your spec as the description

### Step 2C: Create Linear Issue

**Alternative:** If your team uses Linear instead of GitHub Issues

1. Navigate to your Linear workspace
2. Click **New issue** (+N)
3. In description field: **Paste** (⌘+V) your exported spec
4. Add title from spec
5. Set project, priority, and assignee
6. Click **Create issue**

## Spec Traceability

### Link Back to Discovery Loop

To maintain bidirectional traceability between your spec and GitHub:

**Manual method (v1.0):**
1. After creating PR/Issue, copy the GitHub URL
2. Return to Discovery Loop Coach
3. Open your spec
4. Add GitHub link in spec metadata or notes field

**Future (v1.1):** Automatic bidirectional linking with GitHub OAuth integration.

### Spec Versioning

If you update your spec after creating PR/Issue:

1. Re-export updated spec from Discovery Loop
2. Navigate to your GitHub PR/Issue
3. Click **Edit** on the description
4. Replace old spec with new spec (paste)
5. Add comment: "Updated spec from Discovery Loop [timestamp]"
6. Save changes

## Tips & Best Practices

### PR Description Best Practices

- **Keep spec at top:** Paste spec first, add PR-specific notes below
- **Add context:** Include "Closes #123" to auto-link issues
- **Use spec as source of truth:** Reference the spec during code review
- **Update spec, not just code:** If requirements change, update Discovery Loop spec first, then re-export

### Team Workflow Patterns

**Pattern 1: Spec → Issue → PR**
1. PM creates spec in Discovery Loop
2. Export to GitHub Issue
3. Engineer references issue during development
4. Engineer creates PR with "Closes #[issue]"
5. Spec is accessible from both issue and PR

**Pattern 2: Spec → PR directly**
1. Engineer creates spec in Discovery Loop while pairing with PM
2. Engineer writes code
3. Export spec directly to PR description
4. Code review references spec acceptance criteria

**Pattern 3: Spec → Linear → GitHub**
1. PM creates spec in Discovery Loop
2. Export to Linear issue for planning
3. Engineer creates GitHub PR with Linear issue link
4. Spec accessible via Linear → GitHub chain

## Keyboard Shortcuts

- **Export spec:** ⌘+E (Mac) / Ctrl+E (Windows)
- **Copy to clipboard:** Automatic on export
- **Create GitHub issue:** G → I (GitHub keyboard shortcuts)
- **Create PR from branch:** G → P (GitHub keyboard shortcuts)

## Troubleshooting

### Clipboard Not Working

**Problem:** Spec not copied to clipboard after export

**Solutions:**
1. Check browser clipboard permissions
2. Use "Download as Markdown" option instead
3. Manually select and copy spec text from preview

### Markdown Formatting Issues

**Problem:** Spec formatting looks wrong in GitHub

**Solutions:**
1. GitHub supports standard Markdown + GFM (GitHub Flavored Markdown)
2. Preview your spec in Discovery Loop before export
3. Use GitHub's Preview tab when editing description
4. Common issues: missing blank lines between sections

### Large Specs

**Problem:** Spec is too long for GitHub description field

**Solutions:**
1. Break spec into multiple issues (by feature area)
2. Use GitHub Gists for full spec, link in description
3. Store spec in repo `/docs` folder, reference in PR/issue
4. Consider if spec is too broad (should it be multiple specs?)
